# dbt-col-lineage — Context for Planning

## What this repo is

A Python package that productizes the `ProvenanceTracer` POC (currently embedded in `salurbal-data-infrastructure`) into a standalone, rigorously tested library for **static SQL column-level lineage tracing** through dbt DAGs.

The package will be importable (`from dbt_col_lineage import ProvenanceTracer`) and expose a CLI entry point (`dbt-col-lineage trace ...`), replacing the ad-hoc script at `admin-api/_context/release/v0.2/features/dbt-col-lineage-query/poc-1/trace_provenance.py`.

## Why this exists

The `refine-dbt-model` skill in `salurbal-data-infrastructure` traces measure columns through a dbt DAG to source freezes to generate provenance metadata. That logic (in `trace_provenance.py`) is:
- Embedded in a deep context folder, not importable as a package
- Untested — no test suite, correctness is verified by manual inspection
- Fragile — sqlglot lineage is non-trivial to get right across SQL patterns (aggregates, unions, CTEs, window functions)

This repo extracts, hardens, and packages it.

## Core dependency: sqlglot

sqlglot is the only viable tool for this use case. There is no equivalent in any other language.

The specific API used: `sqlglot.lineage.lineage(column, sql, dialect="duckdb")` — static column-level lineage from compiled DuckDB SQL. No Rust/Go/Java library does this. `sqlparser-rs` parses but has no lineage API. Apache Calcite needs a runtime/JDBC. sqlglot is the only engine with both DuckDB dialect support and a built-in column lineage walk.

This is a **Python package, full stop.** The CLI it exposes is still Python under the hood (argparse/click entry point), same as dbt itself.

## The testing strategy: demo upstream dbt

The package will ship with a **fixture dbt project** — a minimal, self-contained dbt project with:
- A set of controlled source definitions (fake source freezes with known metadata)
- A set of intermediate models with known SQL patterns (CTEs, unions, aggregates, window functions, joins)
- A compiled `manifest.json` (generated by `dbt compile` on the fixture project)

Tests assert that `ProvenanceTracer.trace(model, column)` returns the exact expected source freezes for each fixture case. This is the only way to validate correctness — unit tests on isolated SQL snippets miss the manifest graph traversal; integration tests against the real dbt project are too slow and environment-dependent.

The fixture project doubles as living documentation of which SQL patterns are supported.

## Current implementation (POC reference)

**File:** `salurbal-data-infrastructure/admin-api/_context/release/v0.2/features/dbt-col-lineage-query/poc-1/trace_provenance.py`

**Key classes/functions:**
- `ProvenanceTracer(manifest, dbt_package)` — initialized with pre-loaded manifest dict
- `.trace(model_name, column)` → `list[dict]` — one row per source freeze
- `walk_depends_on_to_sources(manifest, start_key)` — DAG subgraph walk
- `extract_table_refs_from_lineage_nodes(nodes)` — extracts `"external"."table"` refs from sqlglot AST nodes
- `_resolve_table_to_node_key(table_name, manifest)` — maps SQL table ref → manifest node key
- `_chain_lineage(start_model, column, manifest, dag)` — chains sqlglot across intermediate models

**Known edge cases the POC handles:**
- Aggregates (COUNT/SUM/etc.) — falls back to proxy column (SALID1/YEAR) to continue traversal
- `read_parquet(...)` references — matched to source nodes via external_location
- Manifest generated by `dbt parse` (no `compiled_code`) — detected and rejected at init
- Missing model nodes — fallback to manifest `depends_on` graph

**Known gaps in the POC (to address in this package):**
- No tests
- SQL comment stripping is naive (`re.sub(r"--.*$", ...)`) — breaks inline strings with `--`
- `_find_proxy_column` is a heuristic (looks for SALID1/YEAR/SALID2 by string match, not AST)
- No handling for models with multiple output columns contributing to one column via COALESCE/CASE
- Fallback to `depends_on` graph is silent — no warning when sqlglot can't resolve

## Interface contract (must stay stable)

The downstream consumer is the `refine-dbt-model` skill in `salurbal-data-infrastructure`, specifically `generate_provenance.py`. The public interface that must remain stable:

```python
from dbt_col_lineage import ProvenanceTracer

tracer = ProvenanceTracer(manifest: dict, dbt_package: str)
rows = tracer.trace(model_name: str, column: str) -> list[dict]
# Each row: {target, column, stage_model, source_freeze, country,
#            dua_uuid, source_raw_data_uuid, dmc_harmonized_data_uuid}
```

CLI (replaces `generate_provenance.py --manifest ... --model ... --output ...`):
```bash
dbt-col-lineage trace \
  --manifest path/to/manifest.json \
  --model model_name \
  --column deaths \
  --output provenance.json
```

## What a planning agent needs to decide

1. **Package structure** — single module vs. submodules; where fixture dbt project lives (`tests/fixtures/dbt_project/`)
2. **Fixture dbt project scope** — which SQL patterns to cover (CTEs, unions, aggregates, window functions, lateral joins); minimum to be useful vs. comprehensive
3. **How to ship the compiled manifest fixture** — commit the compiled `manifest.json` to the repo (simplest, no dbt dependency in CI) vs. generate it in CI via `dbt compile`
4. **CLI design** — argparse vs. click; whether to expose `trace` as a subcommand or a top-level command
5. **Error taxonomy** — distinguish `ManifestError` (wrong manifest type), `ModelNotFoundError`, `ColumnNotTraceable` from each other rather than bare `ValueError`/`RuntimeError`
6. **Publish target** — PyPI (public) vs. private index (internal only); affects naming and CI

## Constraints

- Manifest must be from `dbt compile` (not `dbt parse`) — `compiled_code` must be present on every model node
- DuckDB dialect only — the target dbt project materializes to DuckDB/parquet via `dbt-duckdb`
- sqlglot version pin matters — lineage API has changed across versions; pin and test
- Python ≥ 3.11 (uses `Path | None` union syntax)
